## Code intelligence map

Static-analysis output, not LLM guessing. Structure here is verified; BEHAVIOR is
not verified until captured.

### Entry points
Every way behavior starts: UI events, scheduled jobs, file watchers, exposed
COM interfaces, DB jobs, other apps calling in.

- <entry point> - <what it triggers> (led-XXXX)

### Modules and dependencies
Internal modules plus every external component the code binds to. Components with
NO source available are out-of-tree behavior and get capability rows below.

- <module/component> - <role; source available? if not, flag OUT-OF-TREE> (led-XXXX)

### Dead-code candidates (two-entry discipline)
"No static references found" is a verified finding (the analyzer run is the
oracle). "Can be retired" stays a separate ASSUMED claim - late binding, COM event
sinks and dynamic dispatch reach code analyzers cannot see. Retirement is decided
by adjudicating-parity, not here.

| Candidate | Analyzer finding (verified) | Retirement claim (assumed) + discriminator |

## Capability inventory

The heart of the artifact. One row per OBSERVABLE BEHAVIOR (not per file/module).

recipe under `.episteme/captures/<CAP-id>/`. Every `assumed` row names its
discriminator: the concrete capture that would verify it.

| CAP-002 | <e.g. a NULL discount is treated as 0% and the order still posts> | <file + procedure> | assumed | code reading: <what the code appears to do> | led-XXXX | <the capture that would verify it> | LANDMINE |

## Quirk register

Captured behaviors that look like bugs users may depend on. Excavation REGISTERS

DECIDES preserve or fix. Never "fix" a quirk here: record what IS.

| Quirk id | CAP it belongs to | The quirk (what the system actually does) | Evidence (ledger id) | Status |
|---|---|---|---|---|
| Q-01 | CAP-002 | <e.g. a NULL discount is treated as 0% and the order still posts> | led-XXXX | awaiting-adjudication |

## Capture gaps and limitations (honesty section)

What could NOT be captured from the running system and why - GUI paths with no
clean I/O tap, code reachable only at year-end, environments unavailable. These
capabilities remain `assumed` no matter how clear the code reads. For each: the
gap, the reason, and the cheapest path to a future capture (UI automation,
instrumentation shim, logic-layer extraction) or an honest "unverifiable, here is
the residual risk".

- <gap> - <why> - <cheapest path to capture, or residual risk> (led-XXXX)

## Handoff to adjudicating-parity

Status check before handoff: every row tagged + ledger-cited; ledger-check exit 0;
`status: ready-for-adjudication`. Adjudication walks the capability table and
decides per row: retire / retain-on-legacy / migrate-as-is / migrate-and-fix /
repurchase (and, per quirk, preserve or fix). Quirk register entries
(`Q-NN`, awaiting-adjudication) and rows still `assumed` with open discriminators
are its first queue.

- Open questions for adjudication: <e.g. Q-03 (CAP-007) - who depends on it?>
